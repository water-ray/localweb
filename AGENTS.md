# Codex 项目规则

## 基本原则

- 默认使用中文沟通、命名说明和文档内容；代码标识符按对应语言和项目惯例处理。
- 先读取并验证上下文，再回答或修改；不要基于未确认信息扩展需求。
- 只处理用户明确要求的范围；如果发现必要的关联清理，和本次变更一起完成。
- 引用项目内容时使用真实文件路径；除非用户要求，不展开无关实现细节。
- 修改同一文件时尽量一次完成，避免把同一个文件拆成多轮零碎修改。

## 修改边界

- 遵循项目现有风格、目录结构和命名方式，避免无必要的重构、搬迁和格式 churn。
- 不删除无关代码、功能或结构；只有用户明确要求或替换旧能力时才移除。
- 移除或替换功能、配置时，同步删除相关代码、类型、API、文档、测试、持久化字段、任务入口和用户可见字符串。
- 不保留旧流程、死代码、弃用字段、兼容分支、重复别名、历史 fallback 或注释掉的代码，除非用户明确要求兼容。
- 如果必须保留兼容层，在同次变更中写清兼容范围、清理触发条件和移除计划。

## 目录约定

- 通用辅助目录统一使用 `scripts/`、`docs/`、`Bin/`、`temp/`。
- 项目脚本默认使用 Python，放入 `scripts/<用途>/`；不要散落在仓库根目录或源码目录。
- 项目说明、设计、使用文档放入 `docs/<项目或主题>/`；根目录 `README.md` 只保留总览和入口。
- 编译、构建和打包后的最终产物输出到 `Bin/<项目>/`。
- 临时文件、下载、截图、UI dump、临时日志和人工测试输出写入仓库内 `temp/`，不要放到根目录或源码目录。
- 新增脚本或文档时优先复用现有子目录；只有确有新类别时再新增一级目录。

## Codex Skills

- 本仓库的项目内 skill 源位于 `.codex/skills/`。
- 当用户请求项目初始化、子项目接入、功能规则沉淀、skill 生成、Git 提交推送或仓库任务时，先读取对应 `.codex/skills/<skill-name>/SKILL.md` 再执行。
- 如果当前 Codex 环境未自动列出项目内 skills，仍按文件路径读取本仓库的 `SKILL.md`；需要全局触发时，再把 `.codex/skills/<skill-name>/` 同步到 `$CODEX_HOME/skills/`。
- 新项目或新设备迁移时，至少保留 `AGENTS.md`、`.codex/skills/`、`docs/`、`scripts/git/repo_tasks.py`、`.vscode/tasks.json` 和 `.gitignore`。
- 新增子项目时，优先创建或更新一个项目 skill；新增稳定功能或复杂工作流时，优先创建或更新一个功能 skill。

## 执行与验证

- 代码修改完成后，优先使用静态校验、语法检查、lint、类型检查或最小必要测试验证。
- 除非用户明确要求或验证确有必要，不自动启动完整服务、执行完整构建或生成大体积产物。
- Git 提交、推送、拉取和远程配置优先通过 `scripts/git/repo_tasks.py` 或 `.vscode/tasks.json` 中的仓库任务执行。
- 执行有覆盖风险的 Git 操作前，必须先展示风险和即将执行的目标；不要绕过交互确认执行强推或破坏性同步。
